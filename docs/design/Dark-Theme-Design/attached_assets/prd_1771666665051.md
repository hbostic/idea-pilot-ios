# Idea Pilot — Backend PRD
**Role:** System of Record (SoR) + rules engine for all clients

---

## 1. Summary

The backend provides canonical data models, validation rules, and APIs for Idea Pilot across all clients (iOS first). It enforces the execution framework (layers, phases, task constraints) while remaining UI-agnostic.

---

## 2. Goals

- Provide secure multi-user data isolation
- Provide canonical Playbook/Task/Phase models
- Enforce business rules (e.g., task size constraints)
- Support offline-first clients via sync-friendly APIs
- Enable independent evolution of clients (API versioning)

Success (MVP):
- iOS client can fully function (with offline cache) using backend APIs
- 99% successful request rate for core flows
- task validation prevents oversized/vague tasks from entering “Now”

---

## 3. Non-Goals (MVP)

- Team collaboration, roles/permissions beyond single-owner
- Complex automation engine (Zapier-style)
- Billing and subscriptions (can be stubbed)
- Full-text search across everything (optional later)

---

## 4. Core Domain Models (Canonical)

### 4.1 User
- id, email, auth provider identifiers
- created_at, updated_at

### 4.2 Playbook
- id, owner_user_id
- title, description (optional)
- phase (Proof/Structure/Repeatability/Growth)
- created_at, updated_at, archived_at (optional)

### 4.3 Sections
Each Playbook has structured sections:
- Vision (rich text / markdown)
- System (rich text / markdown)
- Build (rich text / markdown)
- BusinessModel (optional rich text / markdown)

Storage options:
- single `playbook_sections` table keyed by (playbook_id, section_type)
- or columns on playbook (simpler MVP, less flexible)

### 4.4 Task
- id, playbook_id, owner_user_id
- title (required, short imperative)
- detail (optional)
- lane: NOW | NEXT | LATER
- estimated_minutes (required; 30–180 in MVP)
- status: OPEN | DONE | CANCELED
- order_index (for stable ordering per lane)
- created_at, updated_at, completed_at (optional)

### 4.5 Capture Item (Inbox)
If you implement a separate capture inbox:
- id, playbook_id, owner_user_id
- content (freeform)
- created_at
- converted_to_task_id (optional)
For MVP you may route capture directly into LATER tasks and skip this entity.

### 4.6 Weekly Cycle
Track weekly planning ritual:
- playbook_id
- week_start_date (ISO week)
- planned_tasks_count
- completed_tasks_count
- notes (optional)

## 5. Business Rules (MVP)

### 5.1 Task Size Enforcement
- `estimated_minutes` must be within 30–180
- If >180: reject with error + suggested split prompt
- If missing: reject (or default to 60 only on client, but backend should validate)

### 5.2 Vague Task Prevention (Lightweight)
Backend can enforce minimal quality by requiring:
- title length >= 6 characters
- no forbidden vague titles (e.g., “work on app”, “do stuff”) via a small denylist
Full NLP “quality scoring” is a later enhancement.

### 5.3 Phase Gating (Soft in MVP)
- Allow tasks in any phase, but store phase state
- Later: enforce constraints (e.g., Growth tasks locked until Repeatability complete)

### 5.4 Ownership & Isolation
- Users can only access their own Playbooks and related objects.

## 6. API Requirements

### 6.1 Authentication
MVP options:
- Hosted auth (Auth0/Clerk/Supabase) OR
- Email+password with JWT+refresh tokens

Requirement:
- short-lived access token
- refresh mechanism
- secure token storage guidance for iOS

### 6.2 REST Endpoints (MVP)
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout` (optional)
- `GET /playbooks`
- `POST /playbooks`
- `GET /playbooks/{id}`
- `PATCH /playbooks/{id}`
- `POST /playbooks/{id}/archive` (optional)

Sections:
- `GET /playbooks/{id}/sections`
- `PUT /playbooks/{id}/sections/{sectionType}`

Tasks:
- `GET /playbooks/{id}/tasks?lane=NOW|NEXT|LATER`
- `POST /playbooks/{id}/tasks`
- `PATCH /tasks/{taskId}`
- `POST /tasks/{taskId}/complete`
- `POST /tasks/reorder` (batch reorder per lane)

Weekly cycle:
- `POST /playbooks/{id}/weekly/plan` (select tasks into NOW)
- `GET /playbooks/{id}/weekly/status`

### 6.3 API Versioning
- Prefix: `/v1/...`
- Backward compatible for all minor changes
- Breaks require `/v2`

---

## 7. Offline-First & Sync Considerations

### 7.1 Client Caching
- iOS caches Playbooks, sections, and tasks locally
- Backend supports `updated_since` or ETag-based fetches

### 7.2 Conflict Strategy (MVP)
- Last-write-wins per record with `updated_at`
- Later enhancement: per-field merge for section text
